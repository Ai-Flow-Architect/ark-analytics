## 0. なぜこの整理書が必要か

サービスページ「納品物（全6点）」に **① GA4 + Search Console → BigQuery 自動連携設定** と明記されている一方、1回目トークルーム（4/22 回答 T1 L360-362）で「Search Console → BQ連携は別途見積もり」と回答しており、**スコープの食い違いに見える状態**になっている。

5/22 MTG で「サービスページに書いてあるのに、なぜ追加見積りなのか」と問われる可能性があるため、何が標準範囲で何が追加範囲かを技術根拠付きで明文化する。

## 1. 結論（境界線）

| 区分 | 内容 | 状態 |
|------|------|------|
| **標準範囲（SVC① に含む）** | ① GA4 → BigQuery 自動連携（events_* エクスポート） | ✅ 実装完了済（4/24納品） |
| **標準範囲（SVC① に含む）** | ② Search Console → BigQuery 一括エクスポート設定（GSC Bulk Export の有効化） | 🔁 接続設定として標準範囲・5/22 MTG までに有効化対応 |
| **追加範囲（別途見積り）** | ③ SC BigQuery 生データの整形・専用テーブル設計（searchdata_* → marts/reports） | 🟠 追加見積り対象 |
| **追加範囲（別途見積り）** | ④ ページ別検索クエリの Looker Studio 可視化チャート構築 | 🟠 追加見積り対象 |
| **追加範囲（別途見積り）** | ⑤ GA4 × Search Console 結合分析（流入キーワード × CV の突合分析） | 🟠 追加見積り対象 |

**一言で**: 「**接続（データを BigQuery に届けるまで）は標準範囲。届いた生データを分析できる形に整形・可視化する部分は追加範囲**」

## 2. 技術的根拠（なぜこの線引きが妥当か）

### 2.1 GA4 → BigQuery（標準・完了）

- GA4 管理画面の「BigQuery のリンク」設定で完結する標準機能
- 納品時点で `analytics_386840839.events_*` に日次エクスポート稼働中
- 工数: 設定のみ（数十分）→ SVC① の主たる履行内容

### 2.2 Search Console → BigQuery 一括エクスポート（標準・接続設定として実施）

- Google Search Console には「**一括データ エクスポート**（Bulk Data Export）」という標準機能がある
- GSC 管理画面 → 設定 → 一括データ エクスポート で、出力先 BigQuery プロジェクトを指定して有効化する
- これは「接続設定」であり、SVC① の文言「Search Console → BigQuery 自動連携設定」に literal に対応する
- → **標準範囲として履行する**（クライアント約束の literal 履行・信頼維持）
- 有効化されると `searchdata_site_impression` / `searchdata_url_impression` テーブルが GSC から日次エクスポートされる

### 2.3 SC 生データの整形・可視化（追加・別途見積り）

GSC が吐き出す BigQuery 生データは **そのままでは分析に使えない**:

- `searchdata_url_impression` は URL × クエリ × デバイス × 国 の巨大な明細テーブル（行数が GA4 events の数十倍規模）
- ページ別・クエリ別の集計、CTR/平均掲載順位の正規化、GA4 セッションデータとの結合キー設計が必要
- Looker Studio で「ページ別検索クエリ」を見せるには専用の marts/reports テーブル + データソース設計が必要

これは SVC① の「接続設定」スコープを超える **データモデリング + 可視化構築の作業** であり、GA4 分析基盤本体（②〜⑥）と同等の工数が発生する。

→ **1回目トークルーム T1 L360-362 の「別途見積もり」回答は、この③④⑤を指していたものとして整合する**

## 3. サービスページ文言との対応表

| SVC 文言 | 本整理書での解釈 | 区分 |
|---------|----------------|------|
| ① GA4 + Search Console → BigQuery **自動連携設定** | GA4・GSC 双方の BigQuery エクスポート**有効化**（接続設定） | ✅ 標準範囲 |
| （SVC に明記なし）SC 生データの分析テーブル化 | searchdata_* の整形・専用テーブル設計 | 🟠 追加範囲 |
| （SVC に明記なし）ページ別検索クエリ可視化 | Looker Studio チャート構築 | 🟠 追加範囲 |

**ポイント**: SVC① の文言は「**自動連携設定**」であり「**検索クエリ分析ダッシュボード**」とは書いていない。連携（=データを BQ に届ける設定）までが①の約束範囲であることを、この対応表で説明できる。

## 4. 5/22 MTG での説明方針

### 想定問答

**Q: サービスページに「Search Console → BigQuery 自動連携」とあるのに、なぜ追加見積り？**

A:
> サービスページ納品物①の「自動連携設定」につきましては、Search Console から BigQuery へデータを届ける**エクスポート有効化**を指しており、こちらは標準範囲として5/22までに対応いたします。
>
> 一方、Search Console の生データは BigQuery 上では URL × クエリ × デバイスの巨大な明細形式で出力されるため、「ページ別の検索クエリを Looker Studio で見たい」というご要望には、GA4 分析基盤本体と同等の専用テーブル設計・可視化構築が別途必要になります。この部分を1回目トークルームで「別途見積もり」とご案内しておりました。
>
> 境界線を整理した資料をご用意しておりますので、5/22 MTG で接続部分（標準）と分析・可視化部分（追加）を分けてご説明いたします。

### 接続部分の標準対応スケジュール

| アクション | 担当 | 期日 |
|-----------|------|------|
| GSC 一括データエクスポート有効化（出力先 `ark-hd-analytics` 指定） | 弊社（要 GSC プロパティ管理者権限の共有） | 5/22 MTG までに対応 or MTG時に画面共有で設定 |
| エクスポート稼働確認（searchdata_* テーブル生成確認） | 弊社（BQ実測）| 有効化後24-48h |

### 追加部分の見積りクロージング

5/21 作成予定の追加見積書ドラフトに以下3点を含める（`/quote-gen` + pricing-master）:
- SC 生データ整形・専用テーブル設計（marts/reports）
- ページ別検索クエリ Looker Studio 可視化
- GA4 × SC 結合分析（流入キーワード × CV）
