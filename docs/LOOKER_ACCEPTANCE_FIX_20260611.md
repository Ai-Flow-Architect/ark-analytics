# Looker Studio 検収指摘 修正手順（2026-06-11・GUI操作 3点）

検収前確認①〜④のうち、Looker Studio側のGUI操作で直すもの。
レポートURL: https://datastudio.google.com/reporting/e26ea2fe-edd9-47d6-8187-dd7c7cd31b8e

> BQ実測による真因（要約）
> - ①ファネル数値差異: Looker側タイルの集計期間に6/9が含まれていた
>   （PDF=6/2〜6/8の202/31/27/24/20はBQと完全一致。Looker表示220/32/31/28/22は
>   BQの6/2〜6/9合計と完全一致＝期間コントロール非連動 or タイル独自期間「今月」）
> - ②期間指定が効かない: 同上（ファネルタイルが期間コントロールに連動していない）
> - ③チャネル棒グラフが6/2以降空: グラフが月次テーブル `channel_kpi_monthly` に紐付き。
>   月次行の日付は毎月1日（2026-06-01）のため、期間開始を6/2以降にすると行ごと範囲外＝空
> - ④問合せ数 20 vs 5: ページ別の表が週次テーブル `page_performance`（週開始=月曜）に紐付き。
>   期間6/2〜6/8では week_start=6/8 の週だけが残り、その週の実績（6/8:3＋6/9:2）＝5件と一致

## 修正A: ファネルタイルを期間コントロールに連動（①②）

対象: ファネル分析ページ＋統合ダッシュボードのファネル系スコアカード/グラフ全部。

1. 対象タイルをクリック → 右パネル「設定」
2. **期間のディメンション** が `report_date` になっているか確認（空なら設定）
3. 「既定の日付範囲」が**「自動」**になっているか確認
   （「カスタム」「今月」になっていると期間コントロールを無視する → これが①②の真因）
4. 集計が **SUM** であること（AVG だと日平均になり、PDFの期間合計と一致しない）
5. 確認: 期間を 2026/06/02〜2026/06/08 にして
   **202 / 31 / 27 / 24 / 20** になればレポートPDFと完全一致＝完了

## 修正B: チャネル分析の棒グラフをdaily化（③）

1. チャネル分析ページのセッション棒グラフを選択
2. データソースを `channel_kpi_monthly` → **`traffic_breakdown_daily`** に変更
3. フィルタ: `dimension_type = channel`
4. ディメンション: `dimension_value`（または日付軸なら `report_date`）
5. 指標: `SUM(sessions)`、期間のディメンション: `report_date`、既定の日付範囲: 自動
6. 確認: 期間 6/2〜6/8 で棒が表示され、合計が **202** になれば完了

## 修正C: ページ別パフォーマンスをdaily化（④）

1. ページ別パフォーマンスの表を選択
2. データソースを `page_performance`（週次）→ **`page_performance_daily`**（日次）に変更
3. 指標はそのまま（`pageviews` / `conversions_from_page` 等・集計はSUM）
4. 期間のディメンション: `report_date`、既定の日付範囲: 自動
5. 確認: 期間 6/2〜6/8 で `conversions_from_page` 合計が **20** になり
   主要分析の問合せ数と一致すれば完了
   （※問合せ20件はすべて完了ページ `/contact/?mode=finish` に計上される仕様。
   「どのページで成約したか」でなく「完了イベントが発生したページ」の集計）
